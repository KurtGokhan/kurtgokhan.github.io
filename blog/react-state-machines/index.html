Replace useEffect Chains with State Machines

Why complex UI state often goes wrong with useEffect, and how a simple state machine pattern can make it predictable.

reactpatternsstate-management

We’ve all been there. A component starts simple — a loading state, a fetch, render the data. Then edge cases pile up: error handling, retries, optimistic updates, race conditions. Before you know it, you’re staring at five useEffect hooks and a dozen boolean flags.

The useEffect Trap

Here’s a pattern I’ve seen in almost every React codebase:

function UserProfile({ userId }: { userId: string }) {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);
  const [retryCount, setRetryCount] = useState(0);

  useEffect(() => {
    setLoading(true);
    setError(null);

    fetch(`/api/users/${userId}`)
      .then(res => {
        if (!res.ok) throw new Error("Failed to fetch");
        return res.json();
      })
      .then(data => {
        setUser(data);
        setLoading(false);
      })
      .catch(err => {
        setError(err);
        setLoading(false);
      });
  }, [userId, retryCount]);

  // Can `loading` be true while `error` is also set?
  // Can `user` be stale from a previous userId?
  // What happens if two fetches race?
}

The problem isn’t useEffect itself — it’s that boolean flags create impossible-to-reason-about combinations. With 3 booleans, you have 8 possible states, but only 4 of them are valid.

The State Machine Alternative

Instead of independent booleans, model your states as a discriminated union:

type State =
  | { status: "idle" }
  | { status: "loading"; userId: string }
  | { status: "success"; userId: string; user: User }
  | { status: "error"; userId: string; error: Error };

type Event =
  | { type: "FETCH"; userId: string }
  | { type: "SUCCESS"; user: User }
  | { type: "ERROR"; error: Error }
  | { type: "RETRY" };

function reducer(state: State, event: Event): State {
  switch (state.status) {
    case "idle":
      if (event.type === "FETCH") {
        return { status: "loading", userId: event.userId };
      }
      return state;

    case "loading":
      if (event.type === "SUCCESS") {
        return { status: "success", userId: state.userId, user: event.user };
      }
      if (event.type === "ERROR") {
        return { status: "error", userId: state.userId, error: event.error };
      }
      return state;

    case "error":
      if (event.type === "RETRY") {
        return { status: "loading", userId: state.userId };
      }
      if (event.type === "FETCH") {
        return { status: "loading", userId: event.userId };
      }
      return state;

    case "success":
      if (event.type === "FETCH") {
        return { status: "loading", userId: event.userId };
      }
      return state;
  }
}

Using It in a Component

function UserProfile({ userId }: { userId: string }) {
  const [state, dispatch] = useReducer(reducer, { status: "idle" });

  useEffect(() => {
    dispatch({ type: "FETCH", userId });
  }, [userId]);

  useEffect(() => {
    if (state.status !== "loading") return;

    const controller = new AbortController();

    fetch(`/api/users/${state.userId}`, { signal: controller.signal })
      .then(res => {
        if (!res.ok) throw new Error("Failed");
        return res.json();
      })
      .then(user => dispatch({ type: "SUCCESS", user }))
      .catch(error => {
        if (!controller.signal.aborted) {
          dispatch({ type: "ERROR", error });
        }
      });

    return () => controller.abort();
  }, [state.status, state.status === "loading" ? state.userId : null]);

  switch (state.status) {
    case "idle":
    case "loading":
      return <Spinner />;
    case "error":
      return (
        <div>
          <p>Error: {state.error.message}</p>
          <button onClick={() => dispatch({ type: "RETRY" })}>Retry</button>
        </div>
      );
    case "success":
      return <div>{state.user.name}</div>;
  }
}

Why This Is Better

  1. Impossible states are impossible. You can’t have loading: true and error: something at the same time. The type system enforces this.

  2. Transitions are explicit. Every state change goes through the reducer. You can see at a glance what transitions are valid from any state.

  3. Race conditions are handled. The userId is captured in the state, so stale responses from a previous user ID won’t overwrite current data.

  4. It’s easy to extend. Adding a “refreshing” state or an optimistic update is just another case in the union — no new boolean combinations to worry about.

The best code isn’t clever code. It’s code where the wrong thing can’t happen rather than code where the wrong thing doesn’t happen.

You don’t need a library for this. useReducer with a discriminated union gives you 80% of what XState offers, at zero dependency cost. Start there and reach for a library only when your state machine outgrows a switch statement.
