Building Type-Safe API Clients in TypeScript

How to leverage TypeScript's type system to build API clients that catch errors at compile time, not in production.

typescriptapidx

If you’ve ever spent time debugging a runtime error caused by a mistyped field name or a wrong response shape, you know the pain. TypeScript can eliminate an entire category of these bugs — but only if you set things up right.

The Problem

Most API clients start like this:

async function getUser(id: string) {
  const res = await fetch(`/api/users/${id}`);
  return res.json(); // Returns `any` — we've lost all type safety
}

The return type is any. Every consumer of this function is flying blind. You might access user.naem instead of user.name and TypeScript won’t say a word.

A Better Approach

Define your API contract as types, then enforce them at the boundary:

interface User {
  id: string;
  name: string;
  email: string;
  role: "admin" | "member" | "viewer";
}

interface ApiRoutes {
  "/api/users/:id": {
    GET: { params: { id: string }; response: User };
  };
  "/api/users": {
    GET: { response: User[] };
    POST: { body: Omit<User, "id">; response: User };
  };
}

Now build a thin client that uses these types:

type RouteParams<T extends string> =
  T extends `${infer _Start}:${infer Param}/${infer Rest}`
    ? { [K in Param | keyof RouteParams<Rest>]: string }
    : T extends `${infer _Start}:${infer Param}`
      ? { [K in Param]: string }
      : Record<string, never>;

async function apiClient<
  Path extends keyof ApiRoutes,
  Method extends keyof ApiRoutes[Path],
>(
  path: Path,
  method: Method,
  options?: {
    params?: RouteParams<Path & string>;
    body?: ApiRoutes[Path][Method] extends { body: infer B } ? B : never;
  },
): Promise<
  ApiRoutes[Path][Method] extends { response: infer R } ? R : never
> {
  let url: string = path;

  if (options?.params) {
    for (const [key, value] of Object.entries(options.params)) {
      url = url.replace(`:${key}`, value);
    }
  }

  const res = await fetch(url, {
    method: method as string,
    headers: options?.body ? { "Content-Type": "application/json" } : {},
    body: options?.body ? JSON.stringify(options.body) : undefined,
  });

  return res.json();
}

The Payoff

Now every call is fully typed — both inputs and outputs:

// ✅ Fully typed — response is `User`
const user = await apiClient("/api/users/:id", "GET", {
  params: { id: "123" },
});

// ✅ TypeScript knows `user.role` is "admin" | "member" | "viewer"
if (user.role === "admin") {
  // ...
}

// ❌ Compile error — "naem" doesn't exist on User
console.log(user.naem);

// ❌ Compile error — missing required `params`
const bad = await apiClient("/api/users/:id", "GET");

Key Takeaway

The best DX improvement you can make is one that prevents mistakes silently. You don’t need a runtime validation library for internal API calls — the type system can do the heavy lifting if you invest a small amount of time upfront.

The pattern above is simplified, but the idea scales. Libraries like tRPC and Hono take this much further, but understanding the underlying mechanics helps you build better abstractions in any codebase.
